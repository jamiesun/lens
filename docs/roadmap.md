# Lens 项目画像与方向

## 项目概述

Lens 面向单个用户，是一个运行在 Chromium 浏览器中的前端 Agent Runtime。它在 Side Panel 中提供对话、页面摘要、动作计划、确认、日志和结果，让用户通过统一插件操作自己的多个业务系统，而不必在每个系统里重复实现整套 Agent。

Lens 以结构化页面能力为主、通用 DOM 理解为兜底。模型负责理解目标并提出工具调用；本地运行时负责校验、授权、确认、执行和记录。模型服务使用用户自己的凭据直接连接，业务操作复用现有系统及其后端，不新增 Lens 服务端。

```text
                           ┌────────────────────┐
                           │ Model Provider API │
                           └─────────▲──────────┘
                                     │ user-owned key
┌────────┐     ┌─────────────────────┴─────────────────────┐
│  User  │ ↔   │ Side Panel UI ↔ Local Agent Runtime       │
└────────┘     │                  ├─ Policy / Confirmation  │
               │                  ├─ Secret Vault           │
               │                  └─ Run State / Audit      │
               └─────────────────────┬─────────────────────┘
                                     │ typed messages
               ┌─────────────────────▼─────────────────────┐
               │ Content Adapter / Semantic Page Snapshot  │
               └─────────────────────┬─────────────────────┘
                                     │ validated bridge
               ┌─────────────────────▼─────────────────────┐
               │ Host Page Protocol / data-agent-* / DOM   │
               └─────────────────────┬─────────────────────┘
                                     │
                           ┌─────────▼──────────┐
                           │ Existing Business │
                           │ Systems and APIs   │
                           └────────────────────┘
```

## 项目画像（目标状态）

- **一次安装，服务多个自有系统。** Agent 交互、模型接入、策略和日志只实现一次；每个业务系统只声明页面语义与自身能力。
- **语义优先，DOM 兜底。** 优先使用系统注册的结构化工具，其次使用稳定的语义标记，最后才分析通用 DOM。页面信息按需、渐进地提供给模型，而不是默认上传整页 HTML。
- **受控执行，而非无限自主。** 模型只能提出符合 Schema 的工具调用。权限、域名、页面状态、风险、确认、超时和审计由运行时强制执行，模型和页面都不能绕过。
- **用户始终掌握写操作。** 读取和普通填表可以自动进行；提交、删除、审批、付款等操作必须按照风险策略获得与具体参数绑定的确认。
- **本地优先且诚实描述安全边界。** 模型密钥加密后保存在本地，解锁材料不跨浏览器会话持久化。页面数据、工具结果和日志默认最小化并脱敏，不把本地加密包装成无法兑现的硬件级安全。
- **可恢复且不重复副作用。** 页面变化、导航、浏览器扩展运行时休眠或任务中断，都不会让过期节点继续执行，也不会因恢复和重试重复产生业务写入。
- **可观察但不过度留存。** 用户能够看到计划、确认项、执行结果和必要的本地轨迹；敏感值不会为了“完整日志”而永久保存。

发生取舍时，优先级依次为：

1. 用户控制、业务正确性和数据安全，高于自动化程度。
2. 稳定协议和可验证执行，高于模型临场猜测能力。
3. 本地简洁和低维护成本，高于企业级集中治理能力。
4. 一个可靠的单步闭环，高于多 Agent、长期记忆和大规模自主编排。

## 当前能力清单

- **Chromium MV3 扩展基础已可运行。** WXT 配置、后台 Service Worker 和 Side Panel 入口分别位于 `wxt.config.ts`、`entrypoints/background.ts` 与 `entrypoints/sidepanel/`。
- **聊天优先的本地 Side Panel 已经建立。** 目标输入、支持 Markdown 与 LaTeX 公式的回复和多轮历史构成主界面；页面摘要、手动表单与工具日志收进页面抽屉，入口位于 `entrypoints/sidepanel/App.tsx`。
- **本地文件附件已经接入聊天输入。** 输入区通过可扩展的“+”入口选择文本、代码与数据文件，运行时以独立的非可信参考数据边界发送给用户配置的模型；单文件、数量和总大小均受限，历史只保留附件元数据。
- **页面语义快照与表单填写已经实现。** 运行时注入的常驻 `entrypoints/page-agent.content.ts` 使用 `src/content/page-observer.ts` 提取可见标题、表单、字段、动作、表格与告警，不返回密码、隐藏字段或实际表单值；无按钮语义但带点击示能（cursor: pointer / onclick）的元素以 `clickable` 动作收录；`src/content/form-controller.ts` 通过原生 setter 填写可见非敏感字段，逐字段返回结果，陈旧快照整体拒绝。
- **受控页面点击已经实现。** `src/content/click-controller.ts` 只按最新快照的 nodeId 派发真实指针事件序列（pointerdown → mousedown → pointerup → mouseup → click，含元素中心坐标）；表单提交按钮与声明为 server-write/destructive/financial 的元素由运行时直接拒绝，隐藏、禁用、脱离文档与陈旧快照均不执行。
- **消息与结果边界已经类型化。** `src/protocol/` 定义快照、填写及页面命令 Schema，`src/background/page-service.ts` 负责活动标签检查、结果校验和权限错误分类。
- **本地模型连接和密钥保管已经实现。** `src/background/secret-vault.ts` 使用 PBKDF2-SHA256 与 AES-GCM 保存用户自己的 API Key，解锁密钥只进入 `storage.session`；模型来源使用可选 host permission，且仅接受 HTTPS 或回环 HTTP 地址。
- **受控 Agent 目标循环已经实现。** `src/background/agent-runtime.ts` 读取当前快照、调用 OpenAI-compatible 模型，并只路由已注册的 `page.snapshot` / `page.form.fill` / `page.click` / `page.screenshot` 工具；Side Panel 展示目标、工具事件和最终回复。
- **本地会话历史已经实现。** `src/sidepanel/history-repository.ts` 通过 IndexedDB 保存最多 30 个会话；关闭并重开 Side Panel 后可以恢复、切换和删除历史，页面快照与工具原始载荷不会进入历史。
- **页面截图与整页长图已经实现。** `src/background/screenshot-service.ts` 支持当前视口捕获和纵向分段拼接；整页截图受高度、段数、像素与字节上限约束，并在成功或失败后恢复滚动位置、滚动行为和临时隐藏的悬浮元素。
- **站点长期授权管理已经实现。** 点击工具栏图标通过 `action.onClicked` 手动打开 Side Panel,使 Chromium 真正授予 activeTab(`openPanelOnActionClick` 路径不会授权,见 crbug.com/40904917);面板已打开时再次点击图标会就地重新授权并自动重扫,而不是关闭面板。`src/sidepanel/site-access.ts` 在页面可读时识别当前站点访问状态,用户可在「页面信息」抽屉把任意 HTTP(S) 站点升级为可撤销的长期 host 权限(逐站点经浏览器弹窗确认,不默认授予任何站点);撤销被浏览器拒绝时会复核真实权限状态,不会谎报已撤销。授权变化通过 `permissions.onAdded` 触发自动重扫恢复。
- **完整 Settings 已经实现。** 右上角齿轮直接切换到 Side Panel 内的设置页，不打开弹窗或新窗口；页面包含「模型与密钥」「运行参数」「站点授权」「关于」四个可键盘切换的标签。运行参数在本地持久化并受共享 schema 硬边界约束，下一次 Agent 运行会读取模型步数、输入上下文和输出 token 上限；站点授权页枚举真实 host 权限，撤销失败后重新读取浏览器权限。
- **当前能力具备自动化证据。** `tests/unit/` 覆盖语义提取、表单控制器、Vault、模型客户端、Agent 循环和截图边界；`tests/e2e/` 在真实 Chromium 扩展中覆盖页面访问、填写回执、历史恢复/删除、截图下载、整页拼接、页面恢复、锁定恢复和完整模型工具循环。

显式 Agent 计划预览、表单提交、私有页面协议、业务 API、确认策略和持久化审计仍未实现，不属于当前能力；页面点击可用但提交与声明高风险的动作一律被运行时拒绝，直到确认策略落地。

## 非目标（铁律）

- 不做面向任意网站的通用自主浏览器代理，也不默认申请 `<all_urls>`、Cookie、历史记录、下载等广泛权限。
- 不使用悬浮球或注入式聊天窗作为主要交互界面。
- 不新增 Lens 服务端、Token Broker、多租户、团队权限、配额或计费系统；现有业务系统后端不受此限制。
- 不提供 `eval JavaScript`、任意脚本、任意 URL 请求或其他绕过工具注册与策略校验的万能接口。
- 不允许模型决定是否需要确认、如何定级风险或是否记录审计；这些都是运行时不可绕过的职责。
- 不默认发送原始完整 DOM、密码字段、Token、隐藏敏感值或无关页面内容给模型。
- 不承诺通用撤销。只有业务能力明确提供补偿动作、必要前置状态和有效期限时，才显示可回滚入口。
- 不永久明文保存 API Key，也不把长期解密密钥与密文共同持久化。
- 不建设多 Agent、长期记忆、无人值守跨站浏览或自动操作大量页面的编排平台。

## 方向与意图

### 统一的本地交互面

用户能够在 Side Panel 中完成目标输入、页面理解、计划审阅、动作确认和结果查看。关闭或重新打开面板不会造成执行状态含混，也不会隐藏仍待确认的高风险动作。

### 渐进式页面理解

插件能够形成精简、可定位且经过敏感信息过滤的页面语义快照，并按需深入表单、表格、告警或选定区域。页面变化会使旧节点身份失效，运行时能够拒绝陈旧操作而不是盲目点击。

### 可见、可控的表单协助

用户可以让 Agent 识别字段、填写内容并读取校验结果。自动填写产生的变化清晰可见；提交及其他外部副作用遵守风险策略，失败不会留下无法解释的半完成状态。

### 跨系统复用的页面协议

自有系统可以通过同一份轻量协议声明页面类型、字段语义、能力、输入输出和结果验证方式。接入新系统不需要复制 Agent UI、推理运行时或安全策略，协议版本差异能够被明确识别和拒绝。

### 受控业务工具

页面能力、现有业务 API 和本地 Agent 能力以结构化工具呈现。每次调用都有明确来源、Schema、权限、风险和结果边界；工具描述与页面内容都被视为输入数据，而不是能够修改运行时规则的指令。

### 本地模型连接与密钥保管

用户能够配置兼容的模型供应商并在浏览器会话中显式解锁。持久化内容不足以单独恢复明文密钥；锁定、解锁失败、密文损坏和供应商错误都有清晰、可恢复的用户反馈。

### 风险控制与本地追踪

运行时能够对观察、本地修改、服务端写入、破坏性和财务动作应用不同策略。确认与具体工具、参数、目标和页面状态绑定且会过期。执行轨迹能够解释发生了什么，同时遵循脱敏和有限留存原则。

### 中断恢复与副作用保护

任务状态能够经受扩展运行时休眠、页面导航、网络失败和用户取消。恢复、重试和重复消息不会造成同一个业务写操作执行两次，无法安全恢复时应明确停止。

## 完成的样子

- 用户安装插件、配置并解锁自己的模型凭据后，可以在允许的业务页面中直接使用 Side Panel，不依赖新增服务端。
- Agent 能够展示可信的当前页面摘要和动作计划，并能按需检查特定表单或表格，而不是上传完整 DOM。
- 用户可以用自然语言填写一张业务表单；字段变化可见，页面校验可读，未经确认不会提交。
- 自有系统通过薄协议接入后，其稳定业务能力优先于 DOM 猜测被发现和调用；新增系统无需重新实现 Agent。
- 无效 Schema、无权限、错误域名、陈旧节点和过期确认都会产生明确失败，且不会留下未声明的外部副作用。
- 高风险操作的确认准确展示工具、目标、参数和预期影响；拒绝确认后不会执行。
- 扩展运行时中断后能够恢复可恢复任务；涉及业务写入的重试不会产生重复记录。
- 本地密钥、页面快照、模型输入和执行日志遵守最小暴露原则，用户能够锁定凭据并清理本地数据。
- 核心业务能力有自动化验证守护，违反项目边界或安全策略的回归能够在合入前被阻止。

## 验收矩阵（业务能力覆盖矩阵）

> 覆盖底线（硬性规定）：
>
> 1. 每个一级功能至少有一条 Happy Path E2E。
> 2. 每个高风险功能至少覆盖一条失败路径。
> 3. 每个涉及权限的功能至少验证两种角色或访问状态。
> 4. 每个会修改系统状态的操作至少验证一次失败后的恢复或回滚。
> 5. 每次新增一级业务功能，必须同步新增对应的 E2E 并更新本矩阵。

Side Panel 对话与计划展示、完整 Settings、当前页面快照、表单填写、受控页面点击、本地模型、Agent 目标循环、运行参数与站点长期授权管理达到覆盖底线；其余项目仍是待补齐的验收缺口。

### 缺口最低期望

- **私有页面协议与工具注册：** 覆盖受信页面、未授权页面、协议版本不兼容和 Schema 非法，失败时不得执行工具。
- **现有业务 API 工具调用：** 覆盖有权与无权状态、服务端失败和重复请求，证明失败不留下部分写入且重试不重复创建数据。
- **风险分级与用户确认：** 覆盖低风险自动执行、高风险阻断、批准、拒绝和确认过期，证明模型与页面不能降低风险。
- **任务恢复与幂等保护：** 覆盖运行时休眠、页面导航、取消和重复消息，证明恢复后不会重复外部副作用。
- **本地审计、回放与清理：** 覆盖成功、失败、拒绝和取消轨迹，并证明敏感参数被脱敏、清理后无法继续读取。
